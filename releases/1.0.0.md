## 🛠️ Source Tools Family

### Self-Aware Code Management

Powerful suite for reading, editing, and managing source code — **including your own implementation**. Perfect for keeping documentation synchronized with code.

### Eight Core Tools

1. **`source_project`** — Switch working directory between projects
   - `switch` — Change to new project path
   - `reset` — Return to original SOURCE_DIR
   - `status` — Show current and original directories

2. **`source_read`** — Browse and search source code
   - `tree` — List all files (respects .gitignore)
   - `read` — Read file content
   - `grep` — Regex search with optional file glob

3. **`source_edit`** — Targeted string replacement (preferred)
   - Exact `old_string` → `new_string` substitution
   - Auto-generated diff preview
   - Requires unique match (context-based)

4. **`source_write`** — Create new files only
   - Full file replacement
   - Auto-creates parent directories
   - Shows diff preview

5. **`source_delete`** — Remove files
   - Safe deletion with confirmation
   - Works with .gitignore rules

6. **`source_run`** — Execute shell commands
   - Runs in project directory
   - Full shell access for builds, tests, etc.

7. **`source_test`** — Run configured test command
   - Uses SOURCE_TEST env var
   - Verify changes work immediately

8. **`source_git`** — Version control operations
   - `status`, `add`, `commit`, `push`, `diff`
   - Safety tiers: read-only (no approval) → local writes → remote (always approve)
   - Blocked: `reset --hard`, `push --force`, `clean -f`, `rebase`

### Workflow Pattern

```
1. source_project (switch if needed)
2. source_read (tree/read/grep to understand)
3. source_edit (apply changes)
4. source_test (verify)
5. source_git (commit + push)
```

### Key Benefits

- **Self-modifying** — Edit your own codebase
- **Documentation sync** — Keep docs in sync with implementation
- **Multi-project** — Switch between projects dynamically
- **Safe operations** — Approval gates for destructive actions
- **Diff previews** — See changes before applying

## 🌐 Web Research System

### Three Search Engines

1. **Tavily** — AI-optimized search API with content extraction
2. **Keiro** — Proprietary search backend with structured results
3. **DuckDuckGo** — Privacy-focused, no API key required

**Engine Configuration**:
- Multiple engines can run in parallel for comprehensive results
- Automatic deduplication across sources
- Failover support when engines fail
- Results merged and ranked by relevance

### Three Fetch Modes

1. **Regular Mode**
   - Standard HTTP requests with custom User-Agent
   - Fastest option for simple pages
   - Limited against JavaScript-heavy sites

2. **Stealth Mode** (Default)
   - Uses `got-scraping` with realistic browser headers
   - Rotates User-Agent strings to mimic Chrome/Linux desktop
   - Bypasses basic bot detection
   - No heavy dependencies

3. **Browser Mode**
   - Full Puppeteer + StealthPlugin integration
   - Executes JavaScript, waits for network idle
   - Handles long-polling sites with graceful timeout fallback
   - Requires Chrome/Chromium installation
   - Best for heavily protected sites

### Configuration

```json
{
  "mode": "stealth",
  "engines": ["keiro", "tavily", "duckduckgo"]
}
```

## 🎯 TaskMaster: Context-Aware Pipelines

### Context Management & Isolation

Advanced task orchestration system optimized for **low-memory local LLMs**:

- **Context Isolation** — Each pipeline task operates in its own context window
- **No Cross-Contamination** — Task outputs don't pollute other task inputs
- **Memory Efficiency** — Constant memory footprint regardless of pipeline length
- **Clean Handoffs** — Structured data passing between tasks

### Why It Matters

Local LLMs (llama.cpp, Ollama) often have:
- **Limited context windows** (4K-32K tokens)
- **Restricted RAM** (context takes significant memory)
- **Slower inference** with larger contexts

**Traditional approach**: Context grows linearly (10,000+ tokens after 20 turns)  
**TaskMaster approach**: Constant ~500 tokens per task

### Pipeline Structure

```
Task 1 (isolated) → extract result → clear context
Task 2 (isolated) → extract result → clear context
Task 3 (isolated) → extract result → clear context
```

### Use Cases

- **Multi-step research** — Search → Fetch → Synthesize (each isolated)
- **Code generation** — Generate → Review → Fix (each isolated)
- **Data analysis** — Load → Analyze → Visualize (each isolated)
- **Wiki building** — Scan → Index file #1 → Index file #2 (each isolated)

## 📊 Precision Mode

### Computation Discipline

- **Vectorized Pandas Only**: No `iterrows()`, `itertuples()`, or row loops
- **Script Limits**: Max 40 lines, one task per script
- **Output Rule**: Print summary OR save file, not both
- **Error Handling**: Diagnostic script before retry

### When to Use Python

- More than 3 numbers
- Aggregations (sum, avg, count, group-by)
- Data with 10+ rows
- Date math, filtering, sorting
- Anything where precision matters

## 🔄 Knowledge Loop: Karpathy-Style Self-Improving Wiki

### The Feedback Loop

Inspired by Andrej Karpathy's LLM wiki methodology — a persistent, interlinked markdown knowledge base that incrementally improves itself:

**1. Ask the Wiki** → Grep `wiki/` for fast answers from compressed knowledge  
**2. Identify Gaps** → Detect missing/stale documentation  
**3. Research & Validate** → `web_search` + `web_fetch` for current info  
**4. Update Source Docs** → `source_edit` authoritative `docs/` files  
**5. Rebuild Wiki** → `wiki_scan` → `wiki_index` loop  
**6. Commit & Sync** → `source_git` add → commit → push

### Workflow Example

```
User: "Document the new TaskMaster context isolation feature"

System:
1. web_search → Research Karpathy methodology
2. source_write → Create docs/TASKMASTER.md
3. wiki_index → Compress to wiki/docs/TASKMASTER.md
4. source_edit → Update releases/1.0.0.md
5. wiki_index → Re-compress release notes
6. git commit → Push to GitHub
```

### Tool Suite

| Tool | Purpose |
|------|---------|
| `wiki_scan` | Discover uncompressed/stale files |
| `wiki_index` | Compress single file to wiki |
| `source_edit` | Update documentation |
| `web_search` | Find external references |
| `source_git` | Version control |

See `docs/WIKI_KNOWLEDGE.md` for complete guide.
